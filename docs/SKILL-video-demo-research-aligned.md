# Demo Video Skill, Research Aligned

A project demo video is not just marketing. It is a proof asset.

Treat it as part of the project’s screening surface. Its job is to reduce doubt fast by helping a reviewer infer:

- the project exists,
- it works,
- it solves a real problem,
- and the author understands what matters.

This skill is aligned to the research in this repository:

- the first 3 seconds are critical,
- proof-first hooks are stronger than long setup sequences,
- short demos work best when they show the result early,
- and the video should reinforce the README rather than replace it.

**Golden rule: lead with the result, not the setup.**

## Workflow

1. Inspect the repository and any available demo assets.
2. Identify the clearest proof of value the project can show.
3. Select one flow only.
4. Build a short proof-first sequence.
5. Keep the clip tight and readable.
6. Place the video where it strengthens the README.

## Inspect The Repository

Before planning the video, look for:

- live deployment links
- screenshots
- GIFs
- previous recordings
- example user flows
- demo data or test data that looks realistic
- features that clearly show value on screen
- README demo section if one exists

Inspect enough to answer:

- what visual result is strongest
- what user flow is easiest to understand
- what can be shown without explanation-heavy setup
- what makes the project look real rather than tutorial-shaped

## Internal Fact Sheet

Extract:

```text
Project name:
One-line problem statement:
Main visible result:
Best user flow to show:
Has live deployment:
Has screenshot assets:
Has GIF/video assets:
Has realistic data:
Has evidence of real users or iteration:
What proves this is not just a tutorial:
Platform target:
Unknowns that require user input:
```

If there is no clear visual result, say so. Do not pretend a weak flow is strong.

## Research-Aligned Heuristics

### 1. The first 3 seconds matter most

The viewer should see the result immediately.

Do not open with:

- setup
- terminal commands
- editor tabs
- folder structure
- logo animation
- talking head

Open with:

- the app already running
- the key action already in motion
- the output visible

### 2. Proof beats explanation

A short demo works when it establishes confidence before context.

The sequence should make the viewer think:

- this is real
- this works
- I understand what it is

### 3. Keep the demo short

For most portfolio projects, target about 20 to 25 seconds.

That is usually enough for:

- result
- one-line context
- one or two actions
- result again

If the project needs more than that, extend only when the complexity is truly visible and justified.

### 4. Show one flow, not everything

A good demo is not feature inventory.

Choose one flow that best proves:

- value
- clarity
- realness
- independent judgment

### 5. Use proof points

If available, include visible proof such as:

- live deployment
- realistic data
- real user interaction
- nontrivial behavior
- evidence of iteration
- visible constraints or trade-offs

## Recommended Sequence

### 1. Result first: 0–3 seconds

Show the project already working.

Good openings:

- search returns useful results
- form submission completes
- chart updates
- workflow finishes
- generated output appears

### 2. One-line context: 3–5 seconds

Add one sentence explaining:

- what the project does
- what problem it solves

Keep it literal and short.

### 3. Core interaction: 5–20 seconds

Show:

- one user task
- one or two actions
- one visible result

This section should make the project feel stronger than a tutorial clone by showing:

- real logic
- real data
- deployment
- iteration
- or nontrivial behavior

### 4. Result again: 20–25 seconds

End by reinforcing the outcome.

Good endings:

- result shown again
- deployed app visible
- realistic task completed
- one concrete proof point on screen

## Recording Rules

- record a clean screen, not a noisy desktop
- zoom the interface if the text is small
- remove unrelated tabs and notifications
- keep cursor movement deliberate
- avoid unreadable UI text
- prefer one clean flow over many rushed ones

## Voiceover And Text Overlays

Voiceover is optional.

For README demos, silent screen capture plus short overlays is often enough.

If you add narration:

- keep it literal
- explain what is happening
- do not oversell
- avoid generic product language

Use overlays only when they reduce uncertainty:

- what the project does
- what the user is seeing
- why the step matters

## What To Avoid

- starting with setup instead of the result
- long intros
- empty motion
- tiny unreadable UI
- too many features in one clip
- no clear outcome
- no visible proof that the project is real
- vanity metrics inside the video

Stars, downloads, and similar metrics belong next to the video in the README, not inside the video.

## Suggested Tools

Practical options aligned with the research trail:

- Arcade for lightweight browser demo recording
- QuickTime on macOS for simple capture
- DaVinci Resolve for editing
- ElevenLabs as optional voiceover support

## Placement

The video should support the README, not float separately from it.

Best options:

- embed a GIF directly in the README
- link to a hosted video
- link to YouTube if needed

The README remains the main surface. The video helps it carry proof faster.

## Output Standard

The final demo plan or output should make the project feel:

- real
- clear
- focused
- and worth another minute of attention

If the demo succeeds, a reviewer should understand the value of the project before they understand every implementation detail.
