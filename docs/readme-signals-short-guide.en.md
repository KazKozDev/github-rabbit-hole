# README Signals: The Short Practical Guide

This is the compressed version of the main report.

If you do not want the full research trail and only care about what to change in a repository, start here.

## What a README needs to do

A README does not need to explain everything at once.

Its first job is to reduce uncertainty for a reviewer and justify continued attention.

That usually means answering four questions quickly:

1. What is this project?
2. Is it real?
3. Does the author seem competent?
4. Is this worth more of my time?

## The strongest signals

The strongest public signals are usually:

- a clear one-sentence project description,
- a live demo, screenshot, or GIF,
- some evidence of real users or real deployment,
- tests and CI where relevant,
- setup clarity,
- `.env.example`,
- meaningful commit history,
- one section on challenges, trade-offs, or lessons learned.

## What weakens a project

Common weak signals:

- tutorial-shaped clones with no original logic,
- vague README language,
- no visible proof that the project works,
- committed secrets,
- no setup instructions,
- commit history that looks careless,
- projects that appear assembled but not examined.

## The clone rule

The most useful distinction is not clone vs. original.

It is:

- replication as the endpoint,
- versus transformation as the next step.

A clone becomes more valuable when it adds:

- original logic,
- live users,
- real iteration,
- unusual constraints,
- clear technical trade-offs.

## The practical minimum

If you only fix five things, fix these:

1. Add one clear sentence saying what the project does.
2. Add a screenshot, GIF, or live demo.
3. Add basic tests and CI if they fit the project.
4. Add `.env.example` and clean setup steps.
5. Add a short “What I learned” or “What I’d do differently” section.
